# Git Commit Skill

Когда пользователь просит сделать коммит — делать по этим правилам без исключений.

## Формат сообщения

```
<тип>: <что сделано, одной строкой>

<детали если нужны — не обязательно>
```

## Типы

| Тип | Когда |
|-----|-------|
| `feat` | добавлена новая функциональность |
| `fix` | исправлен баг |
| `style` | CSS, визуальные изменения без логики |
| `refactor` | переписан код без изменения поведения |
| `content` | обновлён текст, изображения, контент |
| `docs` | изменены документация, CLAUDE.md, README |
| `chore` | конфиги, зависимости, рутина |

## Правила написания

- Писать на русском если проект русскоязычный, на английском если англоязычный
- Первая строка — максимум 72 символа
- Описывать ЧТО и ЗАЧЕМ, не КАК
- Не писать «исправлен баг» — писать какой именно и где
- Не писать «обновлён файл» — это ничего не говорит

## Плохо → Хорошо

```
fix bug                        → fix: фильтр байок не скрывал статьи при переключении
update styles                  → style: карточки байок — янтарный тег и отступы текста
changed some files             → feat: добавлена секция байок в блог с 5 карточками
```

## После коммита

После `git commit` всегда делать `git push` — иначе изменения остаются только локально и не видны в GitHub.

## Что проверить перед коммитом

- Не попали ли в staged случайные файлы (`.env`, временные файлы, `node_modules`)
- Коммит атомарный — одна логическая единица изменений
- Если изменений много и они разные по смыслу — разбить на несколько коммитов
